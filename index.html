A Brief History of Swift:

Swift programming language was first introduced at Apple’s 2014 Worldwide Developers Conference.

“Initially, it was really just me messing around and nobody knew about it because it wasn’t anything to know about. But eventually, it got a little bit more serious […] So I started talking to my management and some of the engineers that were working on Clang, and they seemed excited about it. We got a couple people working on it part-time and I convinced my manager that it was interesting enough that we could have a couple of people work on it.” – Chris Lattner, January, 2017

The initial reactions and reviews of swift were not great, but not terrible (2015ish). A lot of developers enjoyed certain aspects of Swift’s simplicity and flexibility. After some fine tuning, it was finally time for Swift 4.0 release. This was a significant release for Swift, as the popularity grew exponentially. “One of the reasons that Swift has become so popular over such a short period is probably the fact that it was made by Apple. When a company with such global name recognition and a multimillion army of loyal followers does something, the chances are it is doomed for success (or at least good press coverage).”

    Lets take a look at the Pros and Cons of Swift

    PROS:

  • Rapid Development Process

    • Swift is known for being a clean and expressive language with simple syntax. It also requires less code to perform certain tasks when compared to C programming. It uses ARC (Automatic Reference Counting) to track the app’s usage and memory. This was a major plus for Swift, as it gave developers more time to spend on other details. One example of this is the Lyft (car riding) app. They decided to re-write their entire app using Swift, and ended up writing it in less than one third of the lines.

  • Easy to scale the product

    • In addition to faster development time, you get a product that is future-proof and can be extended with new features as needed. Thus, Swift projects are typically easier to scale. The fact that Apple is more likely to support Swift than Objective-C should also get serious consideration for long-term investment.

  • Better safety and performance

    • One of the main focuses of Swift programming language is its speed and performance. Since Swift was designed and created so recently, they were able to incorporate some pretty neat details. Swift’s initial release claimed a performance increase of 40% when compared to Objective-C.

  • Decreased memory footprint

    • Swift’s ability to use dynamic libraries greatly decreased its memory footprint. When developing, using other open source libraries happens a lot. Now, these libraries can either be static or dynamic (shared). Swift incorporated these dynamic libraries, which helped increase size and load time. “Static libraries are locked into code at the time you compiled them, become the part of your executable file, thus increasing its size and load time. They also can’t be automatically updated since they’re stuck in the version you compiled at. Dynamic libraries, on the other hand, exist outside of your code and are uploaded only when needed. Static libraries need to have copies in all files of your program while dynamic ones only need one.”

  • Automatic memory management

    • Swift’s interesting and automatic memory management system is very well down. Using Automatic Memory Counting (ARC), Swift is able to add a garbage collector function that was actually not introduced to iOS. Most programming languages use the garbage collectors to delete class instances that aren’t needed. While this is very useful for decreasing the memory footprint, it adds load on to the CPU. Before ARC was introduced developers had to constantly and manually manage their classes. Which, as a developer, is something that can become quite tedious at points, but nevertheless, is an extremely important skill to know. ARC is a great feature of Swift, and is something that the developers can truly appreciate. It allows developers to increase theirs apps performance without causing lag to the memory or CPU.

  • Vibrant open source community and learnability

    • Apple’s Senior VP of Software Engineering, Craig Federighi, mentioned, “We saw open sourcing as a critical element to make Swift reach its potential to be the language, the major language for the next 20 years of programming in our industry.” Swift, being such a young language, has major potential. One strong aspect on its side, is the its open source community and learnability. With help from Apple and even IBM, Swift has quickly gained a vast and vibrant open source community. Swift’s potential is endless, but things to be executed in an efficient and strategic manner.

    CONS:

  • Young Language

    • Now, this can be looked at as Pro if only looking at the potential of the language. For this very moment, a young language typically means lots of bugs and fixes. Languages needs decades of fine tuning, and will still never be 100% perfect. Swift needs to mature before it can recognized as top programming language. We will just have to wait and see. Also, Swift has a limited number of libraries and tools (as many are still being developed). There are many resources and tools dedicated to versions of Swift that are useless until newer releases. Overall, yes, Swift is an extremely young language with many fixes and releases to come, but the potential still remains enormous.

  • Backwards Compatibility Issues

    • One huge issue that many developers have expressed are issues with backwards compatibility with newer versions of the language. This forced developer’s to have to complete rewrite their Swift programs when a new version became available. Xcode is a great tool that every developer’s using Mac book should have. Now, it was supposed to help developers update their Swift code to a newer version, but didn’t really fix all the issues

  • Limited Talent Pool

    • Yes, Swift is a great new and upcoming programming language, but it just doesn’t compare to other open source languages. Apple does express that people are demanding more from Swift, and people are trying to put as much effort at possible into learning it, but can be problematic, which leads to developers giving up on learning the language.

  • Poor interoperability with third-party tools and IDEs

    • This is one of major complaints I have come across when researching problems that developers have had with Swift. Using tools is Swift is a major problem, and are causing major problems for developers. Xcode, Apple’s IDE, and what they recommend to write and compile if using Swift, lags in tool support for Swift. This is a huge problem that needs to be fixed or they will lose all their develooprs.

    Lets take a look at some Sorting Algorithms in Swift:

    Sorting algorithms Swift are fairly straightforward and are quite similar to many other popular programming languages. Lets take a look at a couple of the more popular sorting algorithms, and how they are implemented in Swift.

    Bubble Sort:

    This algorithm nowadays has just an educational meaning because it’s always slow (with always I mean it’s slow in the best or worst case), it’s O(n²) in time complexity. This means that it needs always n² steps to converge to the solution (i.e. input N = 50, number of steps to reach the solution = 1000). This is an in-place algorithm because it doesn’t need any external storage to perform the sorting.(From the Blog: Fundamentals of sorting algorithms in Swift)

    Insertion Sort:

    Insertion sort is another interesting sortig algorithm that is acutally used pretty often (more then I would have expected). To understand how insertion sort really wokrs take a look at the following animation. We will then dive into the actual Swift code of how insertion sort is implemented. The easiest way to think about insertion sort is by thinking of a deck of cards. Insertion sort is similar to how a human would sort a deck of cards. Starting with first one and finding a spot among rest to hold it, and m=then moving onto the second card, and so on.

    Lets take a look at the image below. Try to see a understand the difference between insertion sort and bubble sort (which was talked about preivously). After getting a good understanding of how insertion sort work, then take a look at the Swift implementation of it:

    Now that we have a solid understanding of how insertion sort works in general, lets take a look at how it is actually implemented in Swift. The main thing to try to understand here, "is that for every iteration the algorithm finds a key index and it looks back to the first one (in the opposite way if the sort is descending!) in order to find a sorted place for the current item. In this way at the end of every iteration we have a sorted array up to the current key index and an unsorted array from the current key index to the end."(From the Blog: Fundamentals of sorting algorithms in Swift)

    Initially the algorithm cycles over the second to last element, then it blocks the current key element. The algorithm looks back until the first element in order to sort the array from 0 to key, and then swap the elements. Also, it is important to note that Insertion sort is O(n²) runtime, but typically works better then Bubble sort. (From the Blog: Fundamentals of sorting algorithms in Swift)

    Picking the right sorting algorithm to use is a whole another question. Here, we have just looked at two of the most common sorting algorithms, Bubble sort and Insertion sort. Both algorithms can be used for numerous applications, it just depends on what it is exactly you are trying to accomplish.       "Type checking is the problem to decide whether a given type can be assigned to a given λ-term. Type inference is the problem to decide whether a given term is typable and, if yes, to determine the most general such type." - Mr. Kurz

      5. If you notice in those files, the copyright text says “Copyright © 2014–2015 Apple Inc.”, whereas the checkin was made in 2010!     6. Protocol-oriented programming

      7. I’ll admit, this one took a bit of time to even begin to sink in. There is a lot out there on protocol-oriented programming (POP), structs and value type semantics, don’t repeat yourself (DRY principle) and then there’s the infamous Crusty talk; so this was an overwhelming topic early on for me, and still continues to be. But things began to click as I dove in and started restructuring my code to use protocols and protocol extensions. By implementing and conforming my classes and structs to protocols (blueprints), and defining protocol extensions (default method implementations), I started seeing patterns of reusability and more modular code, and watched as my Massive View Controllers shrunk in size becoming more manageable. A great side-effect was that I was leaning towards composition as opposed to inheritance, so I wasn’t bloating my objects by super-classing and inheriting properties and methods I didn’t need." (From Nathan Chen's article on Swift Programming)
